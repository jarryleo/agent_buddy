# Test file audit

Generated by reading every test file in `test/` and cross-checking imports / behaviour against `lib/`.

## Summary

- **72 test files** inspected (70 in `test/`, 2 in `test/manual/`).
- ~70 files are load-bearing and should be kept.
- 1 file has clearly redundant test cases (`reminders_tool_defaults_test.dart`).
- 1 test case in `chat_retry_wire_contract_test.dart` is a near-duplicate of a test in `api_service_orchestrator_error_test.dart` (different code paths, kept - see Ambiguous).
- The intentional smoke test (`widget_test.dart` -> `ApiService constructs`) is called out but kept, because AGENTS.md explicitly documents it as the one allowed smoke test.
- No test file imports a `lib/` symbol that no longer exists. No dead-feature file was found.

## Recommended for DELETION

### test/reminders_tool_defaults_test.dart
- **Reason:** Category 3 (duplicate). The `'ToolBase.isEnabledByDefault default'` test in this file walks the entire `ToolRegistry` and asserts every builtin tool defaults to on, with `reminders` + `google_sheet` as the exception. The same assertion is already covered by the `'fresh install: every other builtin tool IS in activeToolIds and enabled (regression guard for the default-on behavior)'` test in `test/settings_provider_reminders_test.dart` lines 76-95, which checks the same thing AFTER `SettingsProvider.load()` runs (i.e. the actual integration path the user hits).
- **Tests inside (if file partially deletable):**
  - `'defaults to true so the existing tools (fetch_web, current_time, ...) keep their opt-out behavior - only reminders and google_sheet (both need one-time setup) opt in explicitly'` (group `'ToolBase.isEnabledByDefault default'`) - covered by `settings_provider_reminders_test.dart` line 76-95.
  - `'isEnabledByDefault is false on every platform where the tool is supported, so the fresh-install and existing-install seeding paths in SettingsProvider both skip it'` - body is `expect(RemindersTool().isEnabledByDefault, isFalse)`, exactly the same assertion as the previous `isEnabledByDefault is false so the picker is...` test in the same file. The test itself even comments "verifying once is enough". Pure redundancy.
- **Keep tests:**
  - `'id is the wire name the Dart side dispatches on'` - pins the wire id (small but unique).
  - `'isEnabledByDefault is false so the picker is the user first interaction with the tool ...'` - pins the constant directly (more foundational than the post-load check; different layer from `settings_provider_reminders_test.dart`).
- **Verdict:** delete the `'ToolBase.isEnabledByDefault default'` test group (1 test) and the second `'isEnabledByDefault is false on every platform ...'` test (1 test) - the file becomes a 2-test smoke-style guard for the `RemindersTool` wire id + default. Alternatively, delete the entire file and rely on `settings_provider_reminders_test.dart`; my recommendation is to keep the two `RemindersTool`-level tests (they pin the constant without dragging in `SettingsProvider` plumbing).

## Ambiguous - needs human judgment

### test/chat_retry_wire_contract_test.dart - the ClientException test
- **Situation:** This test (lines 22-70) asserts that a `ClientException` thrown by the underlying `http.Client` reaches the chat UI as an `error` event containing the `ClientException` substring. The test passes `messages` but **no `onToolCall`** to `ApiService.streamChat`, so it exercises the **direct** streaming path.
- **Overlap:** `test/api_service_orchestrator_error_test.dart` asserts essentially the same thing - same `ClientException`, same expected error-text substrings - but with `onToolCall: (_) async => '{}'`, exercising the **orchestrator** path. Production always calls with a non-null `onToolCall`, so the orchestrator test is the higher-traffic path.
- **Verdict:** Two valid choices, depending on how strict "duplicate" should be:
  1. **Keep both** - they exercise different code paths inside `ApiService.streamChat` (direct vs orchestrator), and removing either leaves a code path uncovered. This is my recommendation.
  2. **Delete the wire-contract test** - if the team is confident the orchestrator path subsumes the direct path error-event emission, the wire-contract test adds little. (Counter-argument: the direct path is the simpler one, and a future refactor that breaks it would slip through silently.)
- The second test in the same file (`'HTTP 4xx errors still surface (non-retryable hard errors)'`) is unique and should be kept either way.

### test/google_sheet_wiring_test.dart vs test/settings_provider_google_sheet_test.dart - partial overlap
- **Situation:** Both files set up a `SettingsProvider` + `GoogleSheetsService` pair and verify state mirroring after a save. `google_sheet_wiring_test.dart` focuses on the `ToolService.googleSheets` instance identity (`identical()` assertion) and the **regression** that proves the old buggy wiring was indeed broken. `settings_provider_google_sheet_test.dart` focuses on the `SettingsProvider` <-> `service.config` mirroring + toggle-gate + listener-leak-after-dispose.
- **Verdict:**
  - `google_sheet_wiring_test.dart` `'ToolService.googleSheets is the SAME instance as the shared one'` test is unique (only file that asserts `identical()`) - keep.
  - `google_sheet_wiring_test.dart` `'regression: when ToolService is built WITHOUT the shared instance ...'` test is unique (only file that exercises the buggy wiring on purpose) - keep.
  - The remaining 3 tests (`'after save: settings.googleSheetConfig reflects the new state ...'`, `'toggle gate decision is consistent ...'`, `'fresh SettingsProvider constructed after a save ...'`) overlap with tests in `settings_provider_google_sheet_test.dart`. Each overlaps but asserts a slightly different angle (the wiring test asserts via `sharedSheets.load()`; the settings_provider test asserts via the storage round-trip).
- My recommendation: **keep both files** - the wiring test is the canonical regression for the "two-instance" bug, and the settings_provider test is the canonical regression for the mirror-after-save / dispose-leak bugs. Removing either would lose a unique scenario.

- **Verdict:** AGENTS.md section "Tests" explicitly documents this as the one allowed smoke test: *"One smoke test exists (`ApiService` constructs) plus service-layer coverage ..."*. Keep.
